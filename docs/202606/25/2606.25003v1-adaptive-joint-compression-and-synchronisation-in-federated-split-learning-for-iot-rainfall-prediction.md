## 摘要
联邦拆分学习（FSL）能够在带宽受限的物联网设备间进行协作训练，但反复的激活和梯度交换造成了通信瓶颈。现有工作要么单独优化激活压缩，要么单独优化同步频率。本文提出了一个面向物联网降雨预测的FSL框架，该框架通过服务器端基于每个客户端EMA平滑的延迟驱动调度器，联合调控激活压缩和同步间隔ρ。该系统在来自11个气象站的逐小时ERA5数据上，通过17种场景的仿真矩阵以及一个在真实广域链路上部署的4种场景的树莓派实际部署进行评估。仿真矩阵验证了调度器在低、高和混合延迟配置文件下的切换能力，而树莓派部署则验证了同一策略选定的高延迟端点。AUPRC在不同配置下仅有微小变化（仿真中为0.6381-0.6484；树莓派上在0.011以内），表明在此设置下，激进的量化和更稀疏的聚合并未实质性降低预测质量。在树莓派上，选定的端点（int8，ρ=3）相对于float32基线，激活上传负载减少了87%，同步流量减少了54%，同时运行时抖动从±688秒降低到±10秒。

## Abstract
Federated split learning (FSL) enables collaborative training across bandwidth-constrained IoT devices, but repeated activation and gradient exchange creates a communication bot-tleneck. Prior work optimises either activation compression or synchronisation frequency in isolation. This paper presents an FSL framework for IoT rainfall prediction that jointly regulates activation compression and the synchronisation interval \r{ho} via a latency driven scheduler on a server with per client EMA smoothing. The system is evaluated on hourly ERA5 data from 11 weather stations through a 17 scenario simulation matrix and a four scenario Raspberry Pi deployment over a real wide-area link. The simulation matrix validates scheduler switching across low, high, and mixed latency profiles, while the Pi deployment validates the high latency endpoint selected by the same policy. AUPRC varies only slightly across configurations (0.6381-0.6484 in simulation; within 0.011 on Pi), indicating that aggressive quantisation and sparser aggregation do not materially degrade predictive quality in this setting. On Pi, the selected endpoint (int8 with rho=3) achieves an 87% reduction in activation upload payload and a 54% reduction in synchronisation traffic relative to the float32 baseline, while reducing runtime jitter from +/-688 s to +/-10 s.

# 论文总结：面向物联网降雨预测的联邦拆分学习中的自适应联合压缩与同步

## 1. 核心问题与整体含义（研究动机和背景）
- **问题**：联邦拆分学习（Federated Split Learning, FSL）能够在不共享原始数据的情况下实现协作训练，但客户端每次前向传播需上传中间激活（smashed data），反向传播需接收梯度，造成严重的通信瓶颈。
- **现有局限**：现有工作要么单独优化激活压缩（如量化、剪枝），要么单独降低同步频率（如增加本地训练轮数），未能同时考虑两者的耦合影响，缺乏在异构网络条件下运行时联合调控的机制。
- **本文目标**：提出一个联合调控激活压缩与同步间隔的FSL框架，通过轻量级服务器端调度器，根据每个客户端的实时延迟信号自适应调整压缩模式和同步频率，以在保持预测性能的同时最大程度降低通信开销。

## 2. 方法论：核心思想、关键技术细节
- **系统架构**：
  - 1个中央服务器 + 11个边缘客户端，通过gRPC通信。
  - 定义了4类RPC：Register（注册）、Forward（激活交换）、Synchronize（FedAvg聚合）、NotifyCompletion（终止）。
  - 通信控制完全位于服务器端，调度策略通过Forward响应下发给客户端。
- **分裂模型设计**：
  - 客户端：两层LSTM编码器（`ClientLSTM`），将48小时×5维特征窗口映射为固定64维激活向量（float32下256字节/步）。
  - 服务器端：双分支MLP预测头（分类 + 回归），回归仅用于辅助正则化。
- **训练协议**：
  - 局部轮次（epoch）定义为10个优化步。同步间隔ρ控制每ρ个局部轮次后执行一次FedAvg聚合。
  - 聚合采用基于局部轮次数和陈旧度的加权FedAvg，支持超时退化为部分聚合。
- **压缩模式**：
  - 三种对称压缩（同用于激活和梯度）：float32（256 B）、float16（128 B）、int8（68 B，含4字节缩放因子）。
- **自适应调度器**：
  - 每个客户端独立的指数移动平均（EMA）延迟估计（平滑因子α=0.2）。
  - 规则映射：延迟 ≤4ms → float32；4-10ms → float16；>10ms → int8。
  - 同步间隔ρ与压缩模式绑定：severity ∈ {0,1,2} 对应 ρ = clip(ρ_base + severity × ρ_step, ρ_min, ρ_max)，实验中 ρ_base=1, ρ_step=1 → ρ ∈ {1,2,3}。
  - 调度器每步执行（O(1)开销），可响应突发延迟变化。

## 3. 实验设计
- **数据集**：
  - 来自Open Meteo Historical Weather API的ERA5再分析数据，覆盖英国纽卡斯尔11个气象站（2015-01-01至2026-03-31）。
  - 每个样本：前48小时（温度、湿度、气压、风速、降雨）预测未来24小时是否降雨（阈值≥0.5mm）。
  - 按时间划分：训练（2015-2023，80%）、验证（2024，8.9%）、测试（2025-2026，11.1%）。
- **仿真实验（17场景）**：
  - 19个云实例（12vCPU, 22GB RAM），所有进程在同一主机上。
  - 三种延迟注入配置：无延迟、低延迟（~8ms）、高延迟（~50ms），以及一种混合配置（4无延迟+4低+3高）。
  - 六种通信策略：float32 ρ=1（基线）、float16 ρ=1、int8 ρ=1、float32 ρ=3、自适应压缩（ρ=1）、联合自适应（压缩+ρ）。
  - 每种场景在3个随机种子（42,52,62）下重复。
- **树莓派部署（4场景）**：
  - 11个Pi 4 Model B（4GB RAM）作为客户端，云服务器（4vCPU,8GB RAM）通过真实广域链路（RTT ~21-24ms）连接。
  - 四种策略：P1（float32,ρ=1）、P2（float16,ρ=1）、P3（float32,ρ=3）、P4（联合自适应）。
  - 注意：由于RTT始终>10ms，P4自动落入int8+ρ=3端点，不演示频繁切换。
- **对比基准**：以float32 ρ=1为基线，对比单独使用float16、int8、ρ=3、自适应压缩、联合自适应。
- **评估指标**：预测质量（AUPRC为主，ROC-AUC、F1为辅）、通信效率（激活上传负载、同步流量）、系统效率（吞吐量、运行时间、延迟、运行时抖动）。
